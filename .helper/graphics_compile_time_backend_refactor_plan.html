Graphics Compile-Time Backend Refactor Plan

Convert the graphics layer from runtime virtual backend interfaces to a compile-time selected backend contract. The goal is to remove the extra virtual dispatch between Core::Graphics and the actual graphics API while preserving a clean path for future Vulkan, Metal, and other backend implementations.

Status: proposal Scope: full graphics abstraction refactor Updated: 2026-05-29 Primary hotspot: command recording

Decision

Refactor the graphics layer as an all-or-nothing architecture change. Do not keep the pure virtual I* interfaces as the final abstraction surface. Select the graphics backend at compile time, expose concrete backend types through stable core aliases, and use concepts/static assertions to enforce the backend contract.

Key rule: concepts validate the backend shape, but selected concrete types carry the abstraction. The optimizer only gets the real benefit when call sites see concrete types.

Current Shape

1 Frontend calls Core::Graphics
2 Graphics reaches IDevice* or ICommandList*
3 Pure virtual interface dispatch
4 Vulkan backend implementation
5 Vulkan API call

The main overhead concern is the hot ICommandList surface: state setting, push constants, draw, dispatch, mesh dispatch, barriers, and related command recording operations.

Target Shape

1 Frontend calls Core::Graphics
2 Graphics stores selected concrete backend
3 Core aliases expose concrete backend types
4 Backend command/resource method
5 Graphics API call

Runtime backend polymorphism disappears from the primitive command path. Future backend support comes from implementing the same compile-time contract and selecting it in the build.

End State

  • core/graphics/common.h contains backend-neutral data: enums, flags, descriptors, state structs, constants, hashing helpers, and shared non-virtual utilities.
  • Runtime pure virtual graphics interfaces are removed from the public graphics hot path: IDevice, ICommandList, resource interfaces, pipeline interfaces, binding interfaces, and shader table interfaces do not remain as final API contracts.
  • Selected backend aliases define public graphics object and handle types, for example Core::Device, Core::CommandList, Core::BufferHandle, and Core::TextureHandle.
  • Core::Graphics owns a concrete backend selected at compile time, not an IGraphicsBackend pointer.
  • Backend method compatibility is enforced with C++ concepts and static_assert checks.
  • Vulkan classes stop being implementations of shared pure virtual interfaces. They become concrete backend contract implementations.
  • Metal later supplies Metal::Device, Metal::CommandList, and resource types satisfying the same contract.

Backend Type Selection

Add one narrow backend-selection header that maps the active build configuration to the selected concrete types. Most engine code should include stable core graphics headers and consume aliases instead of touching backend namespaces directly. 

#if defined(NWB_GRAPHICS_API_VULKAN)
#include "vulkan/backend.h"

NWB_CORE_BEGIN

namespace GraphicsBackend{
    using Backend = Vulkan::BackendContext;
    using Device = Vulkan::Device;
    using CommandList = Vulkan::CommandList;
    using Buffer = Vulkan::Buffer;
    using Texture = Vulkan::Texture;
};

NWB_CORE_END
#endif

Contract Validation

Concepts should live near the shared graphics contract. Their purpose is to catch backend drift at compile time with useful errors. They are not the runtime abstraction layer. 

template<typename T>
concept GraphicsCommandListApi = requires(
    T& commandList,
    const GraphicsState& graphicsState,
    const DrawArguments& drawArgs
){
    commandList.open();
    commandList.close();
    commandList.setGraphicsState(graphicsState);
    commandList.draw(drawArgs);
    commandList.dispatch(1, 1, 1);
    commandList.dispatchMesh(1, 1, 1);
};

static_assert(GraphicsCommandListApi<GraphicsBackend::CommandList>);

Migration Phases

Phase 1: Prepare the Shared Contract

  • Split common.h responsibilities into shared graphics data and backend-selected type aliases.
  • Add a backend-selection header, for example core/graphics/backend_types.h.
  • Add concept headers for device, command list, resources, pipelines, binding sets, descriptor tables, and ray tracing shader tables.
  • Keep current behavior compiling while the contract headers are introduced.
  • Update Visual Studio source lists for every new header/source file.

Phase 2: Convert Resource Ownership and Handles

  • Replace interface handles such as RefCountPtr<ITexture> with selected concrete handles such as RefCountPtr<GraphicsBackend::Texture>.
  • Make concrete backend resource ref-counting non-virtual where possible.
  • Preserve existing arena deleter and lifetime behavior.
  • Convert resources in dependency order: heap, buffer, texture, staging texture, sampler, shader, framebuffer, binding layout, binding set, descriptor table, pipelines, acceleration structures.

Phase 3: Convert Device and Backend Context

  • Change Graphics ownership from IGraphicsBackend to selected GraphicsBackend::Backend.
  • Change Graphics::getDevice() to return the selected concrete device type.
  • Move backend factory selection from runtime factory function to compile-time backend construction.
  • Remove virtual lifecycle dispatch for backend initialization, swapchain management, presentation, and live-object reporting.

Phase 4: Convert Command Lists

  • Convert all command-list creation and submission signatures to concrete selected types.
  • Remove virtual dispatch from command recording calls: barriers, clears, copies, uploads, state setting, push constants, draw, dispatch, mesh dispatch, ray tracing dispatch, and GPU marker calls.
  • Keep command-list resource retention semantics intact, especially binding-set and shader-table retention until GPU completion.
  • Assert the command-list concept once the Vulkan command list is fully converted.

Phase 5: Remove Interface Remnants

  • Delete final I* graphics object interfaces that no longer define the public contract.
  • Remove virtual and override from Vulkan classes where inheritance is no longer needed.
  • Remove bridge casts and helper functions that existed only to cross the old virtual boundary.
  • Rename remaining objects if needed so the final API no longer suggests runtime interface polymorphism.

Phase 6: Validation and Performance Pass

  • Build all supported configurations.
  • Run graphics, scene, mesh rendering, resource upload, ray tracing, and Vulkan feature tests.
  • Capture command-recording CPU profiles before and after the refactor.
  • Verify symbols or compiler output for direct calls in hot command-list paths where practical.

Non-Goals

  • Do not add a new runtime plugin-style graphics backend system.
  • Do not preserve virtual I* interfaces as compatibility aliases in the final shape.
  • Do not reintroduce material-facing VS + PS fallback paths while touching renderer submission code.
  • Do not move project-owned BxDF/material policy into engine asset paths as part of this refactor.
  • Do not change shader authoring language support. Shader source remains Slang-only.

Required Invariants

  • Graphics still owns a required backend for its lifetime.
  • getDevice() remains an invariant path, not a nullable per-frame defensive check path.
  • Graphics pipeline cache keys still include framebuffer/render-target compatibility.
  • Vulkan command buffers still retain GPU-visible binding resources until GPU completion.
  • GPU marker labels and shader entry-point text preserve exact spelling/case.

Risk Register

Risk Impact Mitigation
Large compile break across graphics consumers High Refactor in dependency order and keep each phase buildable before moving to the next.
Resource lifetime regression after removing interface handles High Convert ownership types first, then command-list retention, then remove old interfaces.
Backend-specific types leak into unrelated core domains Medium Require code outside backend folders to consume selected core aliases, not Vulkan:: names directly.
Concept checks become too broad or unreadable Medium Split concepts by responsibility: device creation, command recording, resources, binding, ray tracing.
Performance win hidden by Vulkan loader dispatch Medium Measure command-recording CPU cost, inlining, and state-cache paths, not only raw Vulkan call cost.

Open Design Decisions

  • Exact file split: whether common.h keeps all data structs or delegates to smaller focused headers.
  • Exact backend selection macro names and build-system ownership.
  • Whether backend type aliases live directly in NWB::Core or under NWB::Core::GraphicsBackend with public convenience aliases.
  • How much of the backend lifecycle context should be named Backend, BackendContext, or GraphicsBackend.

Done Means

  • No hot graphics command path calls through ICommandList or equivalent pure virtual interfaces.
  • No device/resource/pipeline public handle depends on runtime backend polymorphism.
  • Vulkan satisfies the compile-time backend concepts.
  • Renderer behavior and asset contracts are unchanged.
  • Graphics tests and representative render scenes pass.
  • Command-recording profile shows the expected reduction in engine-side dispatch overhead.
