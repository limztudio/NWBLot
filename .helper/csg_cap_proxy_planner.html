CSG Cap Proxy Planner

Replace dynamic CPU cap remeshing with analytic cutter-surface proxy rendering. The goal is less overdraw than an OBB proxy, better behavior for moving cutters, and a path that can work with skinned receivers without CPU skinning cap triangles.

Current Situation

CPU source

Walk receiver cap source triangles and evaluate the cutter SDF at triangle vertices.

->

CPU topology

Extract sign-change segments, connect loops, ear-clip them into cap triangles.

->

GPU draw

Upload frame-local cap vertices and draw them as opaque or transparent cap geometry.

Why Change

  • Moving cutters require cap topology regeneration every frame.
  • GPU-skinned receivers do not naturally expose deformed triangles to CPU cap generation.
  • Large receiver triangles need subdivision to find interior cutter contours.
  • CPU triangulation quality and robustness become a rendering concern.

Preferred Direction

  • Draw shape-specific cutter surface proxies instead of OBBs.
  • Generate proxy patches procedurally in mesh shader for built-in shapes.
  • Use receiver depth, stencil, or mask data to restrict visible cap pixels.
  • Keep SDF discard only as a cleanup rule, not the primary shape limiter.

Implementation Phases

Phase 1

Introduce cap proxy draw data

Add a frame-local cap proxy item containing receiver index, cutter index, receiver group, pass type, shape type, and bounds. Keep the existing CPU cap path beside it until the proxy path is validated.

Phase 2

Built-in procedural proxies

Add mesh shader variants for plane, box, sphere, and capsule. Sphere/capsule patch density should be chosen from projected radius so the cap stays smooth without a dense stored mesh.

Phase 3

Receiver mask contract

Define the rule that makes proxy pixels appear only inside receiver openings. Candidate inputs are receiver depth, receiver id/stencil, and CSG clip state. This is the most important correctness step.

Phase 4

Skinned receiver validation

Verify caps on GPU-skinned receivers without CPU deformed triangle extraction. The cap proxy should follow the cutter and receiver masking instead of requiring CPU-remeshed skin data.

Proxy Shape Plan

Shape Proxy geometry Notes
Plane One quad or bounded grid Clip to receiver/cutter bounds to avoid an infinite plane proxy.
Box Six quads Low overdraw and stable normals. Can cull faces outside receiver bounds.
Sphere Procedural latitude/longitude or octahedral patches Choose segment count from projected radius for smooth circular caps.
Capsule Cylinder band plus two hemispheres Patch density can match sphere quality. Emit only visible bands when bounds permit.
Custom SDF Project-provided proxy module Fallback can be no cap or coarse OBB until the project supplies a proxy.

Open Decisions

  • Depth/stencil mask format for receiver openings.
  • Whether proxy cap lighting reuses receiver material or a dedicated cap material.
  • Transparent AVBOIT cap ordering and accumulation rules.
  • Fallback behavior for unsupported custom SDF caps.

Validation Checklist

  1. Static cube receiver with rotating capsule cutter has no leaks outside the receiver.
  2. Plane, box, sphere, and capsule cutters match existing CPU cap appearance.
  3. Cutters moving every frame do not allocate or triangulate CPU cap vertices.
  4. Skinned receiver test shows caps following the visible skinned surface mask.
  5. Opaque, transparent AVBOIT, depth-only, and selection passes use consistent clip rules.

Keep the CPU cap path as a debug/reference backend until the proxy path passes all built-in shape tests.
