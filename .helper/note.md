# NWBLot Notes

Updated: 2026-07-08

## Important Rules

1. `IDevice` is an essential object in `Graphics`. Do not add runtime null checks for local `IDevice* device` retrievals in this layer.
2. Use `getDevice()` and rely on invariant/assert behavior instead of defensive null handling that adds overhead and hides invalid states.
3. Asset-to-asset and component-to-asset bindings must use typed `Core::Assets::AssetRef<T>`, not raw `Name` or string forms.
4. Graphics pipeline caches must include framebuffer/render-target compatibility in the cache key; material/shader identity alone is not enough when pipeline creation depends on framebuffer info.
5. Basic built-in mesh should live in `.nwb` metadata payloads, not as hardcoded vertex/index arrays inside the cooker.
6. If an asset invariant is format-wide, validate it in the asset/codec layer, not only in one cooker path.
7. Vulkan command buffers must retain bound `IBindingSet` objects until GPU completion; descriptor tables and descriptor-heap ranges are GPU-visible lifetime, not temporary CPU-side state.
8. Explicit textual config tokens like shader `stage` and `target_profile` should fail on unknown values; do not silently fall back to a different backend/language. Per-asset shader compiler selectors are no longer accepted metadata.
9. Vulkan/SPIR-V shader entry points need exact case-sensitive text at pipeline creation time; use `Name` only for identity/cache lookup, and preserve the exact entry-point string separately.
10. GPU debug markers / Aftermath event labels are debug text, not identity keys; keep them as exact strings instead of `Name`.
11. Strip UTF-8 BOM on shader source/include text before compilation or include scanning; otherwise the first token/include can fail on BOM-prefixed files.
12. Do not store unused case-sensitive shader entry-point metadata in hash-only archive/index formats; keep only the lookup keys and exact text actually required at the consuming API boundary.
13. Shader define names and variant signatures are exact case-sensitive shader text; do not route them through `Name` or `ACompactString`, or distinct variants can collide.
14. Short canonical shader metadata tokens like `stage` and `target_profile` are good `ACompactString` candidates; keep `AString` for unbounded or exact-text fields instead.
15. The ECS renderer material contract is shader-driven only: prefer `MeshShader + PS` on mesh-capable hardware, otherwise use `CS + PS`; do not add a material-facing `VS + PS` fallback path back into this renderer.
16. If a mesh-capable device and material both provide the mesh path, failures creating or using that mesh path are errors to fix, not reasons to silently fall back to compute.
17. Keep mesh and compute-emulation renderer resources path-specific: mesh path bindings stay read-only (`SRV` mesh buffers + push constants), while compute-emulation owns the UAV vertex expansion buffer and its internal raster bridge resources.
18. The compute-emulation path still intentionally uses the internal `engine/graphics/mesh/emulation_vs` bridge and the device-manager render-pass integration; those are runtime requirements, not leftovers from the removed material-facing `VS + PS` path.
19. ECS renderer material assets expose only `mesh` and `ps` stages. Do not add a user-authored `cs` or `task` stage for this path; the cooker derives the internal compute-emulation stage from the mesh shader source automatically.
20. In the ECS renderer mesh pass, once material/framebuffer render-path selection is cached, bucket draw work by path first and run path-specific submission loops; do not keep branching on render path inside the hot per-draw submission section.
21. `Core::Filesystem::VolumeSession::pushDataDeferred()` copies payload bytes into the staged volume immediately; caller-local temporary buffers only need to stay alive for the duration of the call, so reuse/scratch-backed staging payload buffers are valid there.
22. The public math API in `global/` lives at global scope, not under `NWB::Math`. `SIMDMatrix` stores row vectors in `v[0..3]`, translation lives in the fourth column (`v[0..2].w`), and `Vector*Transform` evaluates `result = M * v`; do not mix row-vector formulas into this layer without an explicit transpose/convention bridge.
23. The math subsystem lives under `global/`, with SIMD component headers under `global/math/`; keep math in the global/domain layer instead of a separate top-level `math/` tree.
24. `global/simplemath.h` owns scalar helper wrappers, `global/simdmath.h` is the aggregate SIMD include, and the implementation lives in `global/math/{type,constant,convert,vector,quaternion,matrix}.h`.
25. The public math layer in `global/` must not depend on an external upstream math checkout at include time; use the checked-in `global/math/` implementation directly.
26. Keep the math surface naming free of `XM*`/`_XM_*` identifiers so upstream branding does not leak into project APIs.
27. Keep math headers domain-split inside `global/`, and expose the public math API at global scope. Do not reintroduce `global/math.h`, `global/matrix_math.h`, `global/sh_math.h`, `compat_math.h`, shim forwarding headers, or the old `Float4x3` / `LoadFloat4x3*` / `StoreFloat4x3*` compatibility surface. Current matrix storage types are `Float34`, `Float44`, and their unaligned forms such as `Float34U`.
28. In the math layer, keep lane extraction, dot helpers, and 4x4 transpose operations backend-local; do not spill SIMD registers to temporary stack arrays unless the API is explicitly storing into caller-provided memory.
29. The current `global/math/` implementation already includes SSE/AVX/FMA and scalar specialization paths; when optimizing this layer, check what is already merged before copying extension code a second time.
30. There is no active core SH module in the current tree. If spherical harmonics support returns, keep it in the math domain as CPU/core routines only; do not reintroduce `global/sh_math.h` or API-specific cubemap projection helpers to core math until a graphics-domain integration is intentionally designed.
31. In `global/math/matrix.h`, do not add extra transposes to `MatrixTranslation*`, axis-rotation builders, or the projection builders just because the matrix rows contain translation in `w`. `Vector3TransformCoord` / `Vector4Transform` evaluate `result = M * v`, and those builders already place translation/basis data in the correct row-storage layout.
32. In `global/math/vector.h`, stream transform helpers receive row-stored `SIMDMatrix` values and build transposed temporaries only as a backend optimization. Keep naming/commentary aligned with that convention so future ports do not accidentally reintroduce row-vector API semantics.
33. ECS mesh rendering gets projection terms from `CameraComponent` through the renderer-owned mesh view buffer. Do not hard-code near/far depth mapping inside material mesh shaders.
34. ECS mesh rendering gets camera view/projection from `TransformComponent + CameraComponent` through the renderer-owned mesh view buffer. Do not couple renderer camera state to input/controller components such as `FpsCameraControllerComponent`.
35. Input controller state such as fly-camera yaw, pitch, speed, and sensitivity is project/runtime state, not a core ECS rendering component. Keep core camera data limited to renderable camera properties.
36. `CameraComponent` stores projection scalar inputs in one `AlignedFloat4Data` lane pack: x = vertical FOV, y = near plane, z = far plane, w = aspect ratio.
37. Use aligned SIMD vector storage for persistent ECS/render-facing vec4 packs (`LightComponent` color+intensity, deformable joint columns, instance GPU data, mesh view matrices). Keep isolated scalar controls scalar when aligning would only add padding/cache cost.
38. `core/ecs` owns ECS infrastructure only. Production scene/content components such as transform, camera, active-camera state, light, and view-basis helpers live in `impl/ecs_scene` under `NWB::Impl::Scene` and target `nwb_ecs_scene`; do not add `core/scene` forwarding shims or aliases back into core.
39. Keep ECS framework type identity domains split by responsibility: component, system, and message type counters belong in `core/ecs/type_id.h`, not in a content component header or one type domain's implementation file.
40. Prefer `AlignedFloat4Data` for GPU-facing vec4 payloads and runtime ECS component packs. Do not rewrite serialized asset/source records or isolated scalar controls just to force SIMD alignment.
41. Do not use project/runtime naming for scene data. The active scene camera entity is `ActiveCameraComponent::camera` in `impl/ecs_scene`, not a `ProjectComponent` or a revived `SceneComponent::mainCamera`.
42. Ray-tracing affine transforms and deformable runtime picking/hit/preview payloads are vec4-lane data. Store them as aligned SIMD lanes, while preserving exact GPU byte sizes and keeping serialized AABB/source/edit records in their compact scalar layouts.
43. Keep tests domain-owned too: `tests/ecs` validates generic ECS infrastructure with test-local components, while scene/content component coverage belongs in `tests/scene`.
44. `VK_EXT_mesh_shader` support for the default ECS renderer requires the Vulkan `meshShader` feature only. Enable `taskShader` opportunistically when supported, but do not require it for the `MS + PS` path.
45. Required or user-requested Vulkan extensions that have known feature structs must preserve their `DeviceExtensionFeature` metadata when moved between optional, required, and enabled extension maps; otherwise `vkCreateDevice` can enable an extension without enabling its required feature bits.
46. `vkGetRayTracingShaderGroupHandlesKHR` writes shader group handles tightly packed by `shaderGroupHandleSize`. Only SBT buffer records use `shaderGroupHandleAlignment`; do not index the cached handle blob with the aligned stride.
47. Shader binding table buffers must be created with `VK_BUFFER_USAGE_SHADER_BINDING_TABLE_BIT_KHR` and device-address usage; the SBT descriptor bit is not implied by CPU write access or by later `vkCmdTraceRaysKHR` use.
48. Vulkan `queryFeatureSupport()` answers must reflect enabled feature structs, not extension names alone. Advanced ray-tracing extensions such as opacity micromap, cluster acceleration structure, and invocation reorder require their feature structs in the `vkCreateDevice` pNext chain.
49. Optional Vulkan feature paths must also require the loaded Volk entry points used by that path. Do not report ray tracing, opacity micromap, cluster acceleration, cooperative vector, or mesh shader support when the required function pointers are unavailable.
50. Vulkan tracked command buffers own deferred acceleration-structure destruction while command work is in flight. Any path that returns a tracked command buffer to the pool must release those tracked references, including skipped or failed submissions.
51. Vulkan BLAS compaction is a cross-submission state machine: keep pending compactions across command-list state clears, poll query results without blocking, and retain the compacted acceleration structure until the copy submission finishes.
52. Vulkan BLAS triangle `RayTracingGeometryDesc::useTransform` requires a GPU-visible transform build-input buffer. Do not silently ignore the transform and build identity geometry.
53. Vulkan ray tracing shader tables own GPU SBT buffers and must retain their parent pipeline; command buffers must retain the shader table while ray dispatch commands are in flight.
54. Generic integer alignment helpers belong in `global/algorithm.h`; Vulkan code should use `AlignUp`, `AlignUpChecked`, or the typed wrappers instead of keeping local duplicate align-up helpers.
55. Vulkan ray tracing arbitrary procedural primitives are represented by AABB build geometry plus an intersection shader. Native spheres/LSS are a separate `VK_NV_ray_tracing_linear_swept_spheres` path and must stay feature-gated through the device feature struct and pipeline create flag.
56. Generic build-configuration helpers belong in `global/compile.h`; do not keep local module copies of simple `NWB_DEBUG` / optimization-mode checks.
57. `Graphics` owns a required backend for its full object lifetime through `NotNullUniquePtr`; lifecycle `destroy()` tears down backend runtime state, not the backend object itself.
58. ECS infrastructure that owns persistent containers must be constructed with an explicit caller-owned arena. Do not add module-local default arenas or default constructors that hide allocator ownership.
59. `utilities/fbx_to_nwb` owns a standalone logger at entry, links `nwb_logclient`, routes non-interactive status/error/list output through `NWB_LOGGER_*`, and keeps prompts/help/pause/logger-init fallback on direct console streams. Its validation/build/write helpers should log failures in-place instead of propagating diagnostic-only `AString& outError` parameters.
60. Current material metadata fields are defined by `impl/assets_material/metadata.h`: `interface`, `shaders`, `shader_variant`, `parameters`, `transparent`, `two_sided`, and `refractive`. Materials must explicitly author `transparent`, `two_sided`, and `refractive`, including zero values. There is no current `asset.alpha` metadata field; alpha belongs in typed material parameters or shader output. Legacy fields such as `compiler`, `default_variant`, and `instance_override` are validation failures.
61. Mesh material typed payloads use `g_NwbMaterialTypedWords` at binding 6 through `impl/assets/graphics/mesh/runtime.slangi` and generated `.bind` accessors. Do not add legacy material binding 5 or `nwbMaterialFind*` lookup paths; binding 5 in the mesh source set remains meshlet descriptor data.
62. `.bind` material interfaces are the authoring-time schema source for typed material fields and blocks. They own block names, field names, field types, defaults, block class attributes (`material_constant` or `material_mutable`), authored struct/instance names, and generated accessor layout; do not derive typed material fields from material instances, shader reflection, or raw Slang globals.
63. Interface-bound material assets must use block-scoped `asset.parameters`, resolve `asset.interface` by stable virtual path, require a non-empty explicit `asset.shader_variant`, fill missing values from interface defaults, and serialize deterministic typed-layout metadata plus packed typed block bytes. Flat parameters, undeclared fields, unsupported types, and missing required defaults are cook failures.
64. Material shaders include `mesh/authoring.slangi` and then the stable `.bind` virtual include for the selected interface. Shader cook generates deterministic Slang-compatible includes under `material_bind_includes`, resolves authored `.bind` includes to generated content without mutating shader source, and tracks both interface and generated include dependencies for checksum invalidation.
65. Runtime material typed storage is split by block class: material-constant bytes are immutable material identity, while material-mutable defaults may be overridden through `MaterialInstanceComponent`. Overrides must target `material_mutable` fields on the matching interface; current per-pass uploads deduplicate constant ranges by material/layout and mutable ranges by byte content/revision, with finer dirty-row uploads remaining an optimization rather than an authoring-contract change.
66. Frame, view, camera, light, and pass values are not material parameters; keep them in existing frame/view/pass buffers.
67. Renderer frame-target resources and texture-bound binding sets must be created in resource/target setup, not lazily from `update()`, `render()`, draw submission, or material pipeline hot paths. CSG interval peel/sample resources are created with deferred targets and renderer resource validation; draw and pipeline code may only consume the existing layouts/binding sets and fail/assert if setup is missing.
68. The logger server is the sole name-symbol resolver; the logger client must not load, serialize, or upload name-symbol tables. `logger/client/module.{cpp,h}` must not `#include <core/common/name_symbols.h>` and must not carry `pickNameSymbolPayload()`, the `/namesym` upload URL/dispatch, `ClientPayloadKind::NameSymbol`, or the related upload bookkeeping fields. Adding a name-symbol dependency back into the client is a regression.
69. Material `refractive` is a dedicated refractive-caster classification flag, separate from `transparent`. The material metadata decides only that boolean; refraction and shadow values come from the material surface hook through `NwbMeshSurface` fields such as `refractionIor`, `shadowAbsorptionTint`, and `renderCoverage`.
70. Shadow and caustic optical physics are engine-owned. Surface hooks supply optical parameters; the trace integrates Fresnel per crossing and Beer-Lambert absorption over the true entry-to-exit volume path. Do not reintroduce per-hit hardcoded thickness or material-schema `transmittance` fields.
71. Runtime ray effects keep hardware and software backends in lockstep by sharing Slang physics helpers and isolating backend differences behind trace hooks. For stochastic caustics, the parity bar is converged-image A/B within tolerance, not per-frame byte identity.
72. Acceleration structures are runtime-built and device-local, never cooked. The no-RT fallback is a GPU-built triangle BVH that mirrors the hardware BLAS/TLAS shape and traces live posed geometry; do not restore the retired SDF shadow fallback or a raster shadow-map subsystem as the compatibility path.
73. Surfel GI is the current dynamic diffuse-GI path. The retired DDGI grid/probe atlases, `gi_probe_*` shaders, `m_giIrradianceAtlas*`, and `nwbGiProbe*` helpers must stay gone. The surfel pool, cell-head table, counter, snapshots, and constants live on `RendererRayTracingState`, not `DeferredFrameTargets`, while lighting reads a resolved screen-space surfel irradiance texture so the pixel shader never reads the writable surfel pool.
74. Surfel GI details that are load-bearing: hash gathering must walk enough cells to cover `ceil(radius / cellSize)`, trace dispatch uses indirect args derived from the live bump-top counter rather than pool capacity, and surfel-to-surfel feedback reads frame-start pool and cell-head snapshots instead of the live buffers being written this frame.
75. Caustics use the light-side photon-splat architecture: directional/spot lights aim photons at refractive instance AABBs, HW/SW producers share the same refraction/Fresnel/Beer-Lambert math, photons splat into R32_UINT fixed-point accumulators, resolve writes caustic irradiance, and deferred lighting adds it pre-tonemap. The producer is gated off when there are no caustic lights or refractive instances.
76. SW BVH node quantization is a measured dead end for the current shadow workload. Keep the float `NwbBvhNode` path; do not reintroduce `NwbBvhNodeQ`, `NwbBvhRefBounds*`, `nwbBvhQ*`, or quantize/dequant mirror bindings as a perf optimization without fresh evidence. The accepted traversal win is front-to-back child ordering plus coalesced triangle-index fetch.
77. Linux opt/fin GPU timing files can contain `Name::c_str()` hash tokens; decode them against `.namesym` data or use logserver output for readable scope names. For stress-test benchmark runs, prefer the Testing runtime directory as the working directory so `cwd/res` wins over any stale parent `__exec/.../res` mirror.

## Scheduler Architecture

1. Keep responsibilities split:
   - `Graphics` layer uses `Alloc::JobSystem` for orchestration-level async jobs and dependencies.
   - Vulkan backend keeps `Alloc::ThreadPool` for fine-grained `parallelFor` style workloads.
2. Do not unify Vulkan internals onto `JobSystem` without measured evidence (oversubscription/contention/scheduling conflict).
3. Favor adapter-based unification later (if needed) over direct hard migration.

## JobSystem Performance Status

1. Applied:
   - Per-job completion signaling (removed global completion wakeup behavior).
   - Batched enqueue path for ready dependent jobs.
   - Replace `std::function` task storage on hot path.
   - Work-first dependent execution (inline one continuation, enqueue remainder).
   - Split `ThreadPool` synchronization domains (`parallelFor` control vs task queue).
   - Reduce per-completion temporary container overhead.

## Project Bootstrap Invariants

1. `CreateInitialProjectWorld` / `DestroyInitialProjectWorld` are strict engine bootstrap APIs for required core world/system setup and teardown.
2. In project runtime code, `m_world` and required core systems (e.g., renderer system) are treated as required invariants after successful startup.
3. Prefer assertion/invariant semantics in project runtime for required core objects; avoid defensive per-frame null checks for them.
4. For project callback instances created after frame init, keep required core objects as non-null members (owner + references) and tear them down in callback destruction before frame destruction.
5. If ownership and non-null access point to the same object (`UniquePtr` + duplicate reference), prefer a single owner member and keep non-null by construction/contract to reduce duplicated state.
6. For required owned pointers that must not be treated as nullable, use the generic `NotNullUniquePtr<T>` wrapper (no `operator bool`) instead of ad-hoc per-class storage wrappers.
7. Keep `IProjectEntryCallbacks` lifecycle/update callbacks minimal (`onStartup()`, `onUpdate(f32)`, `onShutdown()`), and use the callback object's stored runtime context instead of passing context every call.

## ECS Runtime Type Safety

1. Keep `World::getSystem<T>()` RTTI-free (`/GR-` compatible) by using explicit `SystemTypeId` matching instead of `dynamic_cast`.

## ECS API Shape

1. `World` is still the ownership/lifecycle source of truth for component storage.
2. Keep `World` component APIs (`add/remove/get/hasComponent`) private and expose them only through the generic `NWB::Core::ECS::Entity` facade.
3. Treat `EntityID` as a lightweight ID and `Entity` as the component operation surface.
4. Do not introduce ad-hoc or project-local ECS wrappers for this role.

## Math Matrix Layout

1. In `global/math/matrix.h`, `SIMDMatrix::v[0..3]` stores rows.
2. Translation lives in the fourth column (`v[0].w`, `v[1].w`, `v[2].w`); `MatrixTranslation*` and `MatrixLookTo*` already produce the storage layout consumed by `Vector*Transform`.
3. `Vector*Transform` evaluates `result = M * v`; if a helper needs row-vector semantics, make the conversion explicit instead of changing the public math convention.
4. Stream transforms in `global/math/vector.h` build transposed temporaries for backend efficiency only; those transposes are implementation details, not a different API convention.
5. Compact affine matrix boundaries are convention-sensitive: `Float34` / `Float34U` are the current 3x4 storage snapshots. Do not add an extra transpose in `LoadFloat` / `StoreFloat` for those types, or translation/basis data will round-trip incorrectly.

## GPU Timing / Perf Measurement

1. Smoke apps are GUI processes with no scrapeable stdout (logger routes to the logserver window), so per-pass GPU timing A/B uses the `NWB_GPU_TIMING_FILE` file sink in `tests/smoke/gpu_pass_timing_probe.h`. The smoke project opts in with `CaptureOptions::GpuTimingOnly()` (`gi_test_project.cpp`); the file is appended per report interval.
2. Scope names are readable text (`Name::c_str()`) in a `dbg` build but hashed in `opt`/`fin`, so decode opt timing dumps against the matching `*.namesym` shipped beside the binary.
3. A/B control-pass methodology: a real per-pass win appears ONLY in the target scope while control passes the change cannot affect (`render.shadow_visibility`, `render.opaque_regular`, `render.caustic_resolve`) stay flat. If those controls drift by the same ~1–3% as the target, it is system-wide GPU variance (clock/thermal state between runs), not a real win.
4. This host's GPU timing measurement floor is ~15 µs; sub-floor per-pass slices (e.g. the surfel-trace SH reduction inside the ~0.14 ms `render.surfel_trace`) are unmeasurable here and read as inconclusive. The 2026-07-08 surfel-trace tree-reduction A/B on `stress_test_smoke` (HW) was such a case: target moved −0.005 ms (~3× below run-to-run noise) while every control moved identically.
5. Scene/scope coverage for surfel-GI and SW-BVH timing: `gi_test` / `stress_test` exercise `render.surfel_trace`. The per-mesh SW BVH rebuild path (`render.sw_bvh_sort` wrapping `bvhBitonicSort` in `buildMeshSwBvh`, `impl/ecs_render/raytrace/rt_swbvh.cpp`) only fires on the software/emulation arm — `stress_test_sw_smoke` / `gi_test_sw_smoke` set `NWB_STRESS_TEST_FORCE_RT_EMULATION=1` — so it is invisible in the HW `stress_test_smoke` run. The sort scope is placed at the call site, not inside `bvhBitonicSort`, to keep the one-shot self-test sort out of the per-frame average.
6. `render.sw_bvh_sort` measured result (2026-07-08, opt, `stress_test_sw_smoke`, 2 runs / AMD BC-250 RADV): the scope fires once per mesh-BVH **rebuild**, not per frame — 9 events per run across ~116 intervals at ~4 fps. Pooled over 19 events: **median 4.074 ms, mean 3.856 ms** (min 0.340 is a partial interval fold; max 4.180). It does NOT amortize into the per-frame average, which is the intended placement; the bitonic-sort collapse (commit `c996a917`) is what this scope isolates from the surrounding `buildMeshSwBvh` rebuild. Decode against the HW `stress_test_smoke.namesym` for all other scopes; `render.sw_bvh_sort` is absent from that namesym (predates the scope) so its hash is computed directly (`global/name.h`: 8 independent FNV-64 lanes from `LANE_SEEDS`, byte = `toLower(\\→/)`). The decoder lives at `.helper/decode_gpu_timing.py`.
7. Cooker SIGTRAP root cause (resolved): the `resource_cooker` SIGTRAP during `stress_test_sw_smoke` asset cook was NOT a mesh-cook worker-thread issue (it reproduced with `NWB_COOK_WORKER_THREADS=0`). The real cause was a **bundled `slangc` SIGSEGV** in its SPIR-V backend, which the cooker's `SlangShaderCompiler::compileVariant` (`impl/assets_shader/cook.cpp`) turns into a debug trap when the child crashes. slangc deterministically SIGSEGVs on a `groupshared` tile combined with a nested `GroupMemoryBarrierWithGroupSync` bitonic network placed inside a **separately-called function** — it compiles the identical body fine when inlined into the entry point. The `LOCAL_TILE` collapse (commit `c996a917`) extracted that body into a callee `nwbBvhSortLocalTile` and has been blocking every cook since; the fix is to inline all three mode bodies into `main` (see item 8). A plain `ninja nwb_stress_test_sw_smoke` (dbg/opt) now cooks and links cleanly. The earlier `PrepareGraphicsVolumeAssets`/`large_mesh_many_meshlets.nwb` attribution and the manual relink workaround are obsolete.
8. Phase B collapse (commit `794612f2`), validated 2026-07-08: `bvhBitonicSort`'s Phase B merge loop splits each `sequenceSize > GROUP_SIZE` into inter-tile GLOBAL steps (`compareDistance >= GROUP_SIZE`, one dispatch each — partners cross tiles) plus ONE `GLOBAL_TAIL` groupshared dispatch for the 8 intra-tile steps (`compareDistance = GROUP_SIZE/2..1` — partners stay in-tile). Win: ~9 dispatches + ~9 UAV barriers per sequenceSize drop to 2 (1 GLOBAL + 1 GLOBAL_TAIL). The ascending flag `(global & sequenceSize)` is uniform per GROUP_SIZE-aligned tile and is now used by BOTH LOCAL_TILE and GLOBAL_TAIL — the earlier Phase A `LOCAL_TILE` used `(local & sequenceSize)`, which is wrong at the final `sequenceSize == GROUP_SIZE` round (the bit sits above the tile's low bits, so tiles must alternate; `(local & GROUP_SIZE)` is uniformly 0 and corrupts multi-tile sorts). The three mode bodies are inlined into `main` (not separate callees) because slangc SIGSEGVs on the groupshared+nested-barrier network in a callee (item 7). **Status: VALIDATED** — dbg self-tests all PASS (`BVH bitonic sort self-test PASSED (1000 elements)`, plus build/refit/scene); opt A/B (`stress_test_sw_smoke`, `NWB_STRESS_TEST_FORCE_RT_EMULATION=1`, 4 runs / AMD BC-250 RADV, 17 nonzero events): `render.sw_bvh_sort` **median 4.074 → 0.652 ms (−84%), mean 3.856 → 0.655 ms**; control passes (`shadow_visibility` ~4.68 ms, `opaque_regular` ~0.80 ms, `caustic_resolve` ~1.00 ms) stayed flat.
9. Bitonic sort re-audit (2026-07-08, post-inline-fix): static audit of all three regimes (LOCAL_TILE / GLOBAL / GLOBAL_TAIL) found NO correctness bug — consistent `nwbBvhSortPairLess` (strict total order, lexicographic (key,payload)), correct `(global & sequenceSize)` ascending test in both LOCAL_TILE and GLOBAL_TAIL, disjoint-pair dedup (`partner > local` in-tile / `partner <= index` cross-tile, no races), correct `GroupMemoryBarrierWithGroupSync` placement, and correct CPU `BvhSortPushConstants` mirror. Empirical re-confirmation: dbg self-tests all PASS at 15:22 (after the inline fix `3556e339` @ 15:19 and the dbg `.vol` recook @ 15:20), with the 1000→1024-element (4-tile) sort exercising the full multi-tile path. Fresh opt `render.sw_bvh_sort` capture (run from the correct cwd `Testing/skinning_culling_benchmark_runtime/opt`, 56 intervals): 4 nonzero events, mean 0.516 ms / median 0.402 ms — consistent with (slightly below) the validated 0.652/0.655 ms. **No sort regression exists.**

12. HW vs SW per-pass GPU timing comparison (2026-07-08, opt, AMD BC-250 RADV; `stress_test_smoke` [HW] vs `stress_test_sw_smoke` [SW/emulation], both 90 s, both from the correct fresh runtime cwd `__cmake/build/linux-clang-x64/Testing/skinning_culling_benchmark_runtime/opt`). **Finding: SW is NOT faster than HW overall — the per-pass ms numbers that look favorable to SW are an artifact of comparing interval-averages diluted by different pass-firing rates.** The correct metric is delivered frame rate, and it tells the real story:
    - **Frame rate decays identically in both arms** under the live stress scene: both start ~40–68 fps and collapse to a steady-state **~6 fps** within ~30 intervals (~15 s). At steady state HW and SW are tied at 6 fps — neither is faster once the scene is loaded.
    - Per-pass nonzero-median GPU-ms (HW→SW): on the ray/trace-heavy passes **HW is decisively faster** — `render.surfel_trace` 0.14→0.60 ms (SW 4.3×), `render.caustic_photons` 0.38→2.86 ms (SW 7.6×), `render.mesh_dispatch` 0.14→1.70 ms (SW 12.2×), `render.shadow_visibility` 3.20→4.68 ms (SW 1.5×), `render.raster` 0.68→0.87 ms. These are exactly the passes hardware RT/blas accelerate.
    - SW only "wins" on rasterization/lighting fills (`render.opaque_regular` 1.32→0.80 ms, `deferred_lighting` 0.22→0.15 ms, several avboit stages) — small absolute numbers, and consistent with HW spending its cycles on the trace passes.
    - **Conclusion: HW is the faster architecture; the two arms converge at the steady-state scene ceiling.** The earlier SW-side note framing of `render.frame` 3.0 ms was an interval-mean inflated by sparse nonzero intervals (only 27/114 nonzero) and is not a valid cross-arm comparison.
