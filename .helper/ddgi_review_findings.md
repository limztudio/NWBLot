## [CRITICAL] Trace probe/ray indexing is broken: only ONE probe is ever traced per frame
impl/assets/graphics/gi/gi_probe_trace_sw_cs.slang:406
BUG: main() uses SV_DispatchThreadID with numthreads(64,1,1), but the CPU dispatches dispatch(DivideUp(activeProbeCount,64),1,1) so dispatchThreadID.y is ALWAYS 0. probeIndex=(dispatchThreadID.y+updateCursor)%probeCount collapses to the constant `updateCursor` for every thread, and rayIndex=dispatchThreadID.y*64+dispatchThreadID.x collapses to dispatchThreadID.x. So group 0's 64 threads all trace the single probe whose linear index == updateCursor (each a different rayIndex 0..63); every thread in groups >0 has rayIndex>=raysPerProbe and early-returns. The intended 'workgroup-per-probe' mapping (probe from SV_GroupID, ray from SV_GroupThreadID) is never realized. The CPU dispatch is also wrong for its own stated intent: DivideUp(activeProbeCount,64) packs 64 probes per group instead of one workgroup per probe (comment at raytracing_system.cpp:6889 says the opposite of the code at :6893).
SYMPTOM: With updateDivisor=4, updateCursor only ever cycles 0..3, so out of 2048 probes exactly probes 0,1,2,3 ever receive any rays; the entire rest of the grid stays uninitialized/black. GI never converges anywhere except a 4-probe corner. A naive 'ray-data readback nonzero' check (U3 verify) still passes because that one probe's column is populated, masking the failure.
FIX: Coordinate both sides for true workgroup-per-probe: CPU dispatch(activeProbeCount,1,1) with numthreads(64,1,1); shader derive probeSlot=SV_GroupID.x, rayIndex=SV_GroupThreadID.x, guard probeSlot<activeProbeCount && rayIndex<raysPerProbe, then map the active slot to a real probe index over the round-robin set, e.g. probeIndex = probeSlot*updateDivisor + cursor with a probeIndex<probeCount guard (and make the shader's cursor match the pushed updateCursor rather than double-filtering via nwbGiProbeActiveThisFrame).
VERDICT: CONFIRMED

## [HIGH] Dominant-light occlusion ray zeroes ALL lights' contribution, not just the dominant light
impl/assets/graphics/gi/gi_probe_trace_sw_cs.slang:381
BUG: Direct lighting from every light is summed into `irradiance` (lines 345-354). When the dominant-light occlusion ray hits, line 381 does `irradiance *= 0.0`, which annihilates the accumulated contribution of ALL lights, including the non-dominant ones. D4 specifies only the dominant light is shadowed; the other lights must stay unshadowed.
SYMPTOM: In any multi-light scene, whenever the single brightest (dominant) light is occluded at a probe-ray hit, the bounce goes fully black even though other visible lights still illuminate that surface — over-darkened indirect light / crushed bounce in shadowed regions, exactly where GI is supposed to add fill.
FIX: Track the dominant light's own contribution separately (accumulate it in a temp during the direct loop, or recompute NdotL*radiance for dominantLightIndex) and subtract only that term when the occlusion ray hits: `if(shadowHit.hit) irradiance -= dominantContribution;` (clamped to >=0). Leave the other lights' contributions intact.

FIX: Bind the SAME atlas the blend reads as history-in: prevFrontIsA = (m_giHistoryFrontIsA != 0u) for both prevIrr and prevDist (6730-6732). Then the trace and blend both SRV-read the front and only the back is written — alias-free and correct.

## [HIGH] Chebyshev visibility weight is NOT cubed (plan non-negotiable violated) → light leaks

The code computes chebWeight = variance/(variance + delta*delta) and uses it directly (line 302 blendedWeight = w*chebWeight) with no cubing. FIX: After computing chebWeight, cube it: chebWeight = chebWeight*chebWeight*chebWeight; (only in the occluded branch, matching RTXGI/Majercik and the plan).
